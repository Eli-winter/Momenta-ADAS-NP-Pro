# 用 Claude Code 搭建 AES 版本管理系统 —— 实战复盘

---

## 一、我们做了什么

**AES 版本管理系统**是莲花山项目全车型（SZ26 / MR26 / QZH / SF / UR）的一站式版本构建管理工具，覆盖从触发 CI 构建、飞书消息通知，到追踪 Pipeline 链接、产物包名、测试计划、PR 状态、分支管理的完整闭环。

整个系统从零搭建，历经 9 个版本迭代，最终实现：

| 功能模块 | 说明 |
|------|------|
| 构建发送 | 填写 Base 版本 + 模块信息，一键触发 CI 构建并向飞书群发送 @IAT-BOT 消息 |
| Pipeline 自动同步 | artifact-skill 直接调 Momenta CI API，构建提交后立刻返回 Pipeline 链接 |
| 产物包名自动填充 | 轮询 EP 平台 pipeline variables API，构建完成后自动写入包名 |
| 测试计划 / 测试报告 | 每条构建记录可关联飞书文档链接、填写分项测试结果 |
| 版本记录 | 独立面板，管理已发布版本的模块快照和构建状态 |
| PR 管理 | 追踪开发 PR、TLM 需求、lhs/r6/主线回流状态 |
| 分支管理 | 记录各模块在用分支及废弃分支 |

**技术栈**：FastAPI 后端 + 单页 HTML 前端，数据持久化到 `backend/data.json`，通过 `feishu_sync` 包操作飞书 Open API。

## 二、从零开始：我们实际上是怎么说的

### 第零步：把手工流程说出来

开始之前，整理了手工操作的痛点：

> 每次有模块更新，需要手动去找 Base 版本号 → 拼接消息格式 → 复制粘贴到飞书群 @IAT-BOT → 等 IAT-BOT 回复后手动复制 Pipeline 链接 → 构建完再去 EP 平台找包名 → 填到 Excel 里

每个车型可能同时有多条在飞的构建记录，各状态全靠人脑追踪，经常会忘更新或记错版本号。**能出错的地方太多了，这正是自动化最值得做的场景。**

### 第一条消息：先把核心功能定下来

```
我需要做一个版本管理系统：
- 侧边栏列出车型（SZ26/MR26/QZH/SF/UR）
- 每个车型有一个构建表单：填 Base 版本、模块（name/branch/commit_id）、备注
- 提交后发飞书消息 @IAT-BOT，格式：
  @IAT-BOT 基于以下产物：{base} 更新以下模块：
  {name} {branch} {commit_id}
  remark:{remark}
- 历史记录表格展示时间、Base 版本、模块、状态
- 数据存到本地 JSON 文件
```

Claude 直接生成了完整的 FastAPI + 单页 HTML 框架，第一次运行就能跑通。

### 第一个大坑：飞书群消息只是纯文本

v0.1.0 发出去的消息里，`@IAT-BOT` 只是一段文字字符，机器人根本没有收到触达。

我的描述：
```
消息发出去了，但 IAT-BOT 没有反应。
飞书群里 @IAT-BOT 显示的是灰色文字，不是蓝色的真实 mention。
```

Claude 探查飞书 Open API 后发现：真正触达机器人需要用 `post` 消息类型 + `at` tag，并且需要先解析 IAT-BOT 的 open_id。

解决方案：封装 `feishu_client.py`，实现：
1. 启动时调用 `resolve_member_open_id()` 解析 IAT-BOT 的 open_id
2. 发消息时使用 `send_post_message_with_at()` 构造真实 @mention

### 第二个大坑：等 IAT-BOT 回复才能拿到 Pipeline 链接

v0.3.0 实现了轮询飞书消息历史来自动抓取 IAT-BOT 的回复 URL，但问题是：

- IAT-BOT 回复有延迟，有时几分钟后才来
- 飞书消息 API 有权限限制，读消息历史需要额外的 app 权限

我的描述：
```
IAT-BOT 有时要等几分钟才回复，而且读群消息老是报权限错误。
能不能绕开 IAT-BOT，直接触发 CI 然后立刻拿到 Pipeline URL？
```

Claude 找到了 artifact-skill：`~/.agents/skills/artifact-skill/scripts/artifact_tools.py` 里的 `create_artifact_by_branch()` 可以直接调 Momenta CI API，返回值里就包含 `task_link`。

这是整个开发过程中最核心的技术转折：**从被动等待 IAT-BOT 回复，变为主动触发 CI 立刻拿到链接**。

### 第三个大坑：构建完成后还需要手动去找产物包名

v0.6.0 的需求来自一个真实痛点：构建成功后，最终的产物包名需要去 EP 平台手动查找。

我的描述：
```
构建完成后，产物包名需要去 EP 平台查，很麻烦。
EP 平台的 pipeline 详情页有没有 API 可以读取产物变量？
```

Claude 分析后发现 EP 平台的 `pipeline/variables/global/list` 接口里有 `__flow1_artifacts` 变量，构建完成时 CI 系统会写入产物包名。

实现了 `get_package_name_by_pipeline_task()`：从 Pipeline URL 提取 task_id → 调 EP API → 读 `__flow1_artifacts` → 自动写入记录。

### 第四步：功能越加越多，前端开始难以维护

到 v0.9.0，每个车型子页面需要承载 4 个独立面板（构建记录 / 版本记录 / PR 管理 / 分支管理），单页 HTML 膨胀到几千行。

我的描述：
```
现在每个车型需要加三个子页面：版本记录、PR 管理、分支管理。
用标签页形式，切换时才加载数据，不要全部一起加载。
```

Claude 设计了懒加载方案：
- 4 个 `.detail-panel` 默认隐藏
- `switchDetailTab()` 在首次切换到某个 tab 时才发起 API 请求
- 每个面板独立维护自己的 `_loaded` 标志

## 三、开发过程的实际节奏

### 第一阶段：先跑通核心链路

不是一上来就把所有功能全写完，而是：

1. 先把构建表单 → 飞书消息发送 → 历史记录这条主链路跑通
2. 确认数据持久化正常、前端展示正确
3. 手动在飞书里验证消息格式

**先通核心链路，再叠功能。** 每个版本只加一件事，验证没问题才继续。

### 第二阶段：自动化替代手工步骤

主链路通了之后，逐步把手工步骤自动化：

- v0.2.0：真实 @mention（替代手工拼接文字）
- v0.3.0：IAT-BOT 回复自动同步（替代手工复制 Pipeline URL）
- v0.4.0：artifact-skill 直接触发（替代等待 IAT-BOT）
- v0.6.0：产物包名自动填充（替代手工去 EP 平台查）

每个版本解决一个手工步骤，进度清晰，验证方便。

### 第三阶段：扩展信息管理能力

核心自动化完成后，扩展为更完整的信息管理平台：

- v0.7.0：测试计划和测试报告（路测人员需求）
- v0.8.0：侧边栏改版，信息层级更清晰
- v0.9.0：版本记录 / PR 管理 / 分支管理三个独立面板

## 四、与 Claude Code 协作的真实技巧

### ✅ 描述现象 + 期望结果，不猜实现方案

```
❌ 飞书消息需要用 post 类型 + at tag，你帮我改一下
✅ 消息发出去了，但 IAT-BOT 没反应，@IAT-BOT 是灰色的。
   我需要让它真正被触达，你看怎么做
```

Claude 会自己找解决方案，你的猜测反而会限制它的选项。

### ✅ 先做最小验证，再推广到全量

```
✅ 先只做一条构建记录的 Pipeline URL 自动填充，验证 OK 后再加其他字段
✅ 先只做一个车型的分支管理面板，样式 OK 后再复制到其他车型
❌ 直接说"把所有功能全做完"
```

每步都能立即验证，出错也只影响当前一步，容易回滚。

### ✅ 报错给完整日志 + 操作上下文

```
✅ 点了提交按钮，浏览器控制台报 503，
   后端日志：Chat not connected: Group '莲花山AES 全能闭环助手' not found
   我的 FEISHU_APP_TOKEN 是正确的，但群名搜不到
❌ 发送失败了，你看看为什么
```

完整日志 + 操作路径，让 Claude 不用猜就能直接定位问题。

### ✅ 把「必须一起做」的操作绑定说清楚

```
✅ 新增一个字段时，models.py、main.py 的端点、前端表格三个地方都要同步改
✅ 每次修改完代码，告诉我需要重启服务器才能生效
```

复杂系统里，遗漏一个地方就会出现奇怪 bug，提前说清楚联动关系可以避免很多来回。

### ✅ 用版本号管理进度，随时知道在哪个阶段

每个功能点独立为一个版本号（v0.1.0、v0.2.0...），写进 CHANGELOG.md。

好处：
- 随时可以回溯某个版本的具体改动
- 新功能出问题时，能快速定位是哪个版本引入的
- 向同事说明系统状态时，有清晰的版本锚点

---

**核心一句话：从手工操作中找痛点，每次只自动化一个步骤，每步都能验证——这比一次性设计完整系统要可靠得多。**
